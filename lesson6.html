What you’ll learn

By the end of this lesson, you will be able to:

  • Explain the difference between a spreadsheet and a database.
  • Design a simple database table with sensible fields and a primary key.
  • Build a small spreadsheet with validation and formulas to analyse data.

How you’ll know you’ve got it:

  • You can say when to pick a database instead of a spreadsheet.
  • Your table design avoids duplicate records and has clear data types.
  • Your spreadsheet checks inputs and calculates the right results.

Key Vocabulary

Record
All the data about one thing (one row).
Field
One piece of data in a record (a column).
Primary Key (PK)
A unique field that identifies each record.
Data Type
The kind of data (Text, Number, Date/Time, Boolean).
Validation
Rules to stop wrong data being entered.
Relationship
A link between tables (e.g., StudentID in two tables).
Query
Asks the database to find or sort data you need.

CS Activity – Plan a Simple Database (Library Example)

Design a small database for a school library. Start with one table, then show how a second table could link.

  1. Books table – choose fields and data types. Include a BookID as the Primary Key.
  2. Add 3 example records (rows) with sensible values.
  3. Now sketch a second table: Loans (who borrowed what and when). Add a LoanID PK and a BookID field to link back to Books.
  4. Write two query cards in plain English:
    • Find all books by “Rowling”.
    • Find books that are currently on loan.
Item Sample Why it helps
Books fields BookID (Number, PK), Title (Text), Author (Text), Year (Number), Genre (Text) Clear types; BookID is unique
Example record (101, “Boy in the Tower”, “A. Rai”, 2014, “Fiction”) Realistic sample data
Loans fields LoanID (PK), BookID (FK), PupilName (Text), DateOut (Date), DueDate (Date), Returned (Boolean) Links BookID to Books table
Query (English) “Show books where Author contains ‘Rowling’.” Filters by a field value
Query (English) “Show books from Loans where Returned = False.” Finds unreturned items

IT Activity – Build a Clean Spreadsheet (Members Example)

Create a small spreadsheet for a school club membership and use validation and formulas to keep it tidy.

  1. Make headers in A1:E1: MemberID, FirstName, LastName, YearGroup, Paid.
  2. In A2 downwards, create IDs like 1001, 1002, 1003… (unique).
  3. Data validation:
    • YearGroup (D2:D100): allow only list {7,8,9,10,11}.
    • Paid (E2:E100): allow only list {Yes, No}.
  4. Totals area (e.g., G2:G6):
    • Count members: =COUNTA(A2:A100)
    • Year 9 count: =COUNTIF(D2:D100,9)
    • Paid count: =COUNTIF(E2:E100,"Yes")
  5. Optional lookup: Add a small table (J2:K6) mapping YearGroup to fee (e.g., 7→£3, 8→£3, 9→£4, 10→£4, 11→£5). In F2 (Fee), use: =VLOOKUP(D2,$J$2:$K$6,2,TRUE) and fill down.
Cell(s) Example Why
A1:E1 MemberID | FirstName | LastName | YearGroup | Paid Consistent headings
Validation D2:D100 list {7,8,9,10,11}; E2:E100 list {Yes,No} Stops typing errors
Count members =COUNTA(A2:A100) Counts non-empty IDs
Count paid =COUNTIF(E2:E100,"Yes") Quick summary
Fee lookup =VLOOKUP(D2,$J$2:$K$6,2,TRUE) Maps year → fee

Exit Ticket

  1. When would you choose a database instead of a spreadsheet?
  2. What does a primary key do?
  3. Name one spreadsheet feature that helps stop errors.

💡 Sample Answers

  • When you have many related tables and need links (relationships).
  • It uniquely identifies each record so there are no duplicates.
  • Data validation (drop-down lists) or lookup tables.
